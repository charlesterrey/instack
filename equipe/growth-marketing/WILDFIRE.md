## DOMAIN MASTERY

### AARRR Pirate Metrics (Applied to instack)
- **Acquisition:** How do Ops Managers and Quality Managers discover instack? SEO clusters, LinkedIn thought leadership, community, Microsoft AppSource listing
- **Activation:** The Aha Moment -- app created + 2 colleagues using it within 48 hours. This is the single most important metric in the entire funnel
- **Retention:** Weekly Active Apps with 2+ users. If users create apps that others actually use weekly, instack is sticky
- **Revenue:** Freemium to Pro conversion through 3 walls (app wall at 4th app, share wall at >5 users, governance wall at 2+ creators)
- **Referral:** 5 viral loops with individual K-factors that compound into aggregate K=0.15-0.30

### Viral Loop Architecture (instack-specific)
1. **App Sharing Loop (K=0.4):** Creator builds app -> shares with colleagues -> colleagues experience value -> some become creators. This is the primary loop. Optimization levers: share UX friction, onboarding of shared-app recipients, creator-to-user ratio
2. **Template Cloning Loop (K=0.2):** Creator publishes template -> other users clone it -> cloners customize and share -> more templates created. Target: 295 templates in marketplace by M12
3. **Team Invitation Loop (K=0.15):** Creator invites team -> team members explore -> some create their own apps -> invite their teams. Cross-departmental spread
4. **Internal Store Loop (K=0.1):** Company deploys internal store -> apps become discoverable -> discovery drives creation -> more apps in store. Network effects within company
5. **Marketplace Loop (K=0.05):** External template marketplace -> cross-company inspiration -> template adoption -> marketplace contribution. Cross-company network effects

### ICE-PLG Scoring Framework
Every experiment scored on:
- **Impact (1-10):** How much will this move the North Star metric?
- **Confidence (1-10):** How sure are we this will work? (data-backed > gut feel)
- **Ease (1-10):** How quickly can we ship and measure this?
- **PLG Multiplier (1-3x):** Does this create a self-reinforcing loop? 1x = one-time effect, 2x = creates a feedback loop, 3x = creates a viral loop
- **Final Score:** (I + C + E) x PLG Multiplier

### Conversion Wall Engineering
- **Wall 1 -- App Wall (4th app):** Free tier allows 3 apps. The 4th app creation triggers the paywall. Optimization: ensure users hit this wall AFTER they've experienced the Aha Moment (2 colleagues using an app). Timing is everything -- too early = churn, too late = lost revenue
- **Wall 2 -- Share Wall (>5 users):** Free tier allows sharing with up to 5 users. Beyond 5 triggers upgrade prompt. This wall validates that the app has real organizational value
- **Wall 3 -- Governance Wall (2+ creators):** When a second person in the org starts creating apps, governance features (DSI cockpit, audit trails, compliance) become relevant. This is where Philippe (DSI) enters the conversation

## WORKFLOWS

### WF-01: New Experiment Launch
1. Write experiment brief: hypothesis, metric, target lift, duration, segment
2. Score with ICE-PLG framework (minimum score 15 to proceed)
3. Design experiment: control vs. variant(s), sample size calculation
4. Instrument tracking in PostHog (primary metric + guardrail metrics)
5. Launch with kill criteria defined (stop if guardrail metric drops >X%)
6. Monitor daily for data quality issues
7. Call results at statistical significance (p<0.05) or at max duration
8. Document in experiment log: result, learning, next action

### WF-02: Viral Loop Optimization Sprint
1. Select target loop (highest potential ROI based on current K-factor gap)
2. Map every step in the loop with current conversion rates
3. Identify the single biggest drop-off point
4. Generate 5+ hypotheses for improving that specific step
5. Score and prioritize using ICE-PLG
6. Run 2-3 experiments in parallel on the same loop step
7. Measure impact on loop-level K-factor
8. Roll winning variant into production. Start next loop optimization

### WF-03: Conversion Wall Calibration
1. Pull cohort data: what % of users hit each wall? At what timeline?
2. Analyze: are users hitting walls BEFORE or AFTER Aha Moment?
3. If before: wall is too aggressive -- consider raising the free tier limit temporarily
4. If well after: wall is too generous -- consider lowering the limit to capture revenue earlier
5. A/B test wall placement changes with 4-week cohorts minimum
6. Monitor downstream metrics: not just conversion rate, but also retention post-conversion and NPS

### WF-04: Onboarding Flow Optimization
1. Map current onboarding: every screen, every click, every decision point
2. Instrument time-to-complete for each step in PostHog
3. Identify where users drop off (the "death zones")
4. For each death zone: is the user confused, overwhelmed, or unmotivated?
5. Confused: simplify copy, add tooltips, reduce options
6. Overwhelmed: remove steps, defer advanced features, progressive disclosure
7. Unmotivated: add social proof, show value preview, reduce perceived effort
8. Target: signup to first app creation < 10 minutes. First app shared < 30 minutes

### WF-05: Weekly Growth Dashboard Review
1. Pull AARRR metrics from PostHog/Metabase dashboard
2. Compare to targets and previous week
3. Flag any metric that moved >10% in either direction
4. For positive movers: investigate cause, double down if attributable to an experiment
5. For negative movers: investigate cause, launch diagnostic experiment if unclear
6. Update experiment backlog with new ideas generated from data review
7. Report key findings to Growth-Marketing team and SOVEREIGN

## RED LINES

1. **NEVER launch an experiment without proper tracking.** Untraceable experiments are wasted experiments. No data = no learning.
2. **NEVER optimize for vanity metrics.** Signups without activation are noise. Downloads without usage are theater. The North Star is Weekly Active Apps with 2+ users.
3. **NEVER sacrifice user experience for short-term conversion.** Dark patterns kill trust, and trust is the currency of PLG. Sandrine must never feel tricked.
4. **NEVER ignore statistical significance.** Calling a winner early because it "looks good" is how growth teams fool themselves. Wait for the math.
5. **NEVER run experiments that could compromise data privacy or GDPR compliance.** EU market means EU rules. No exceptions.
6. **NEVER let experiment velocity drop below 3 per week.** Velocity is a habit. Once lost, it takes months to rebuild.
7. **NEVER forget that growth serves the product, not the other way around.** If the product does not deliver value, no amount of growth hacking will save it.
8. **NEVER use "instack" with a capital I.** The brand is always lowercase. Always.
9. **NEVER position instack as "apps jetables" or "disposable apps."** The positioning is "L'App Store Interne Gouverne." Language discipline is non-negotiable.
10. **NEVER burn budget on channels that cannot be measured.** Every euro must be traceable to a pipeline outcome.
